No modelo 1:1, cada tarefa cria uma thread real no sistema operacional. Isso funciona bem quando ha poucas tarefas, pois cada thread e criada e executada diretamente. Porem, quando o numero de threads aumenta, o sistema passa a gastar tempo criando, gerenciando e finalizando cada uma delas. Esse demora de criacao comeca a pesar, deixando o programa mais lento de acordo mais threads sao criadas.

No modelo N:M, e criado um numero fixo de threads reais. Essas threads continuam rodando e vao pegando as tarefas uma por uma ate terminar todas. Dessa forma, o programa nao precisa criar uma thread nova para cada tarefa, o que evita o custo extra de criacao e destruicao. Isso faz com que o modelo seja mais eficiente quando ha muitas tarefas.

Durante os testes:
Com poucas threads (1 a 5), o modelo 1:1 foi um pouco mais rapido, pois o custo no modelo N:M e maior do que o custo de criar poucas threads.
A partir de 10 tarefas, o modelo N:M foi mais eficiente, pois nao havia mais gasto criando novas threads.
Em cenarios com 50, 100, 250, 500 e 1000 tarefas, o N:M continuou mais rapido, pois o numero de threads reais continuou o mesmo e apenas as tarefas mudavam.

A diferenca entre os modelos nao foi tao grande devido ao computador usado no teste (Apple M4), que tem um sistema operacional e um processador muito eficientes no gerenciamento de multiplas threads. Em sistemas menos otimizados, como Windows, a diferenca seria muito maior.

Sendo assim, o modelo N:M e mais recomendado para aplicacoes que precisam lidar com muitas tarefas ao mesmo tempo, pois escala melhor e utiliza melhor os recursos do sistema, ja o 1:1 para sistemas mais simples que nao demandam muitas tarefas ao mesmo tempo do processador.
